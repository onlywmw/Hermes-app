## 重新定位

> 运行 tab 不是系统监控——是**设备集群的控制面板。**

王墨微有三台设备：

- **平板**（这台）— 跑 MOV 本体 + DeepSeek API 调用

**他不是在监控"系统进程"——他是在管理"设备集群"。**

## 3. 每张设备卡片里有什么

### 设备基本信息

| 信息 | 来源 | 为什么 |
|------|------|--------|
| 设备名 + 图标 | 用户自己命名 | 知道是哪台 |
| 状态点 🟢🟡🔴⚫ | 心跳检测 | 一眼知道在不在线 |
| 核心硬件 | 设备上报 | PC: GPU 型号。手机: 存储空间 |
| IP 地址 | 局域网发现 | 知道怎么连 |
| 运行时长 | 设备上报 | 稳定跑了多久 |

### 设备上跑的模型

每个模型一行：名称 + 状态 + 今天调用次数 + 平均延迟。

多个模型时，按调用频率排序。用得最多的在最上面。

### 设备上正在执行的任务

- **定时任务 (Cron)**：名称 + 状态 + 上次执行时间
- **训练任务**：名称 + 进度条 + 预计剩余时间
- **一次性任务**：名称 + 状态

任务完成后 → 保留 1 小时 → 自动从卡片上消失（任务历史在归档里看）。

## 4. 设备互联是怎么实现的

```
平板 (MOV)
  │
  ├─ 局域网发现 (mDNS / 手动输入 IP)
  │    └─ 找到 PC → 配对码确认 → 连接建立
  │
  └─ 每台设备上跑一个轻量 Agent
       ├─ 暴露 HTTP API: /status /tasks /models
       ├─ 每 10 秒向 MOV 发心跳
       └─ MOV 运行 tab 聚合展示
```

### Agent 是什么

- 一个超轻的服务（Python 脚本或 Go 二进制）
- 运行在 PC / 手机 / 服务器上
- 暴露 3 个端点：`/status` `/tasks` `/models`
- MOV 平板通过局域网 HTTP 轮询（每 10 秒）

### 配对流程

```
PC 上: ./mov-agent --name "我的GPU电脑"
  → 终端打印: 配对码 4821

MOV 平板上: 运行 tab → [＋ 连接新设备]
  → 输入 IP: 192.168.1.5
  → 输入配对码: 4821
  → 连接成功 → PC 卡片出现
```

## 5. 第一期做什么

| 优先级 | 内容 | 说明 |
|--------|------|------|
| 🔴 现在 | **本机卡片**。平板自身的状态。DeepSeek/Claude/等模型的实时调用统计。Cron 任务。 | 不需要联网，不需要 Agent。就是现在运行 tab 应该有的东西——但用设备卡片的形式展示。 |
| 🟡 以后 | **局域网 Agent**。PC 上跑 mov-agent。平板能发现并连接。 | 需要写 Agent。 |
| 🔵 以后 | **远程设备**。不在同一个局域网也能连。 | 需要中继服务器。 |

**第一期只做本机。** 但卡片结构设计成支持多设备——以后加新设备时，只是多一张卡片。

## 8. 和 DESIGN_STORAGE 的关系

设备卡片上显示的模型调用统计，来自 `StatsCollector` 的本地数据（今天调了多少次、平均延迟多少）。不额外存——实时聚合。

Cron 任务来自 SharedPreferences + CronManager 的 lastRun/lastStatus。
