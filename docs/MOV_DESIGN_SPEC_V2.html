01

四条原则

V1 的"终端技术感"靠三样东西撑起来：网格底纹、全等宽字体、金色铺陈。 V2 逐一替换，但保留 MOV 的专业底色 —— 不做成另一个 iOS 设置页。

PRINCIPLE 01

内容是唯一的装饰

删掉全局网格底纹与装饰性光斑。背景只剩一层极淡的暖光晕，让房间卡片、消息气泡、文件行自己成为视觉主体。留白按 8pt 网格放大一档。

body 背景: 纯色 --bg + 单光晕 · 卡片间距 8 → 12px
PRINCIPLE 02

毛玻璃只给"壳层"

导航栏、底部 Tab、Sheet、看板触发条 —— 这些"悬浮在内容之上"的层用毛玻璃；内容卡片一律实心白。玻璃表达层级，不表达风格。

glass 层: nbar / bnav / sheet / cfoot / btrigger · 卡片: --panel 实心
PRINCIPLE 03

等宽字体退居数据层

V1 所有文字都是 JetBrains Mono，这是终端感的主要来源。V2 中正文、标题、按钮全部回归系统 Sans；Mono 只出现在它该出现的地方：时间戳、文件名、cron 表达式、徽章、延迟数值。

Sans: 标题/正文/按钮 · Mono: 时间/文件名/cron/徽章/数值
PRINCIPLE 04

金色是落款，不是底漆

信使金保留为唯一主色，但用法收敛：选中态、运行中、AI 产出标记、主操作。大面积金色背景（acc-tint 铺满的卡片）改为白底 + 金色细节。黑（--ink-1）承担"确认/发送"的重量感。

金: 选中/运行/产出/主操作 · 黑: 用户气泡/发送/FAB
MONO 保留区 (MOV 的专业签名)
  • 时间戳 / 延迟值 — 14:32 · 210ms
  • 文件名 / 路径 — PRD-v1.md
  • cron 表达式 / pid / 内存 — 30 8 * * *
  • 徽章 / 标签 / 分组头 — 团队 · 产出
  • 代码块 / 工具调用详情
SANS 接管区 (Apple 的可读性)
  • 大标题 — 房间 / 运行 (30px · 800 · -0.03em)
  • 消息正文 — 13px · 行高 1.62
  • 房间名 / 任务名 / 模型名
  • 按钮文字 — 600 字重，不再 letter-spacing
  • Sheet 标题与说明文案

03

场景一 · 和 AI 团队协作

四个设计问题的回答。右侧机器可交互：点底部 Tab 切换视图，点"讨论 / 文件"切换子页。

人和 AI 的对话混在一起，怎么一眼分清谁说了什么？
三层识别叠加，不靠单一手段：位置（用户右、AI 左）＋头像色（每个 AI 固定取自 --av-1~6 的色块，首字glyph）＋名字行（名字 + 角色徽章 + 时间戳）。用户气泡保持深黑，是"我的声音"的锚点；AI 气泡统一白底浮起。不同 AI 之间只靠头像色区分，不给气泡加彩色左边框 —— 克制。
名字行: Sans 11px 650 名字 + Mono 8.5px 角色徽章(金底) + Mono 8.5px 时间 气泡: 圆角 17px · 指向侧塌缩为 5px · 用户黑底 / AI 白底+sh-1
AI 产出了文件，怎么在聊天里展示、怎么在文件区组织？
聊天里用文件卡：白卡 + 深色 glyph 块 + Mono 文件名 + 产出者/大小 + 两个胶囊操作（打开 / 版本）。左缘 --ok-dot 绿边表示"AI 产出物"，与工具调用的金边区分。文件区沿用四分区（产出/资料/归档/模板），产出区文件同样带绿缘标记，版本按钮收进右侧胶囊。
文件卡: r-md 14px · 左缘 3px --ok-dot · 操作按钮 acc-tint-2 胶囊 工具调用: 左缘 3px --acc-live · 详情区 --code-bg
多个项目房间，列表怎么展示才不混乱？
放弃"一屏散卡片"，改用 Apple grouped list：白色大圆角组容器按状态分组（进行中 / 已归档），组内以 line-soft 细分隔。每行信息三层：头像叠堆（成员一目了然）→ 名字+最后消息预览 → 右侧 Mono 时间 + 未读金点。Council 房间挂深色"团队"小标，与单聊房间区分。
组容器: r-lg 20px 白底 · 组头 Mono 9.5px 大写间距 行高 ~64px · 按压态 panel-2 · 未读点带 acc-tint 光晕环
第一次接触"拉 AI 团队"，怎么让用户理解并做出选择？
新建 Sheet 第二步的两个选项卡，各配一个文字图形化的迷你示意：一对一 = 「你 ⇄ MOV」两个圆；团队 = 三个彩色头像 → 「投票箱」。不解释抽象概念，直接画出来。选中态用金环 + 浅金底，下方补一行 Mono 小字说明适合场景。默认选中"一对一"，降低首次决策成本。
选项卡: r-md · 选中 2px 金环 + acc-tint-2 示意 glyph: 头像 --av-* 圆 + 箭头字符 · 不引入图标库

场景二 · 个人使用

沉浸与切换的平衡，以及自带应用和自建应用的视觉区分。看板 mockup 在上一场景的机器里（切到"看板"Tab）。

怎么在沉浸使用和切换应用之间找到平衡？
沿用 V2 架构的全屏 iframe + 悬浮触发条，这是对的，只升级质感：应用占满全屏（音乐页是深色沉浸底），触发条做成深色毛玻璃胶囊，3 秒无操作自动沉底消失，触碰屏幕下缘唤醒 —— 沉浸时它不存在，需要时一抬指就到。面板改为毛玻璃大圆角 Sheet，从底部弹出，带 grabber。
触发条: --glass-dark + --blur · 高 40px · 胶囊 · 3s 自动隐藏 面板: --glass + r-xl 26px · grabber 36×4.5 · 弹出 --t-med 弹簧
自带应用和用户添加的应用，视觉上怎么区分？
用图标块的"底"来区分身份，不用角标轰炸：自带应用 = 深色实心块 + 金字 glyph（延续 MOV 的深色 glyph 语言，像系统原生）；自建应用 = 白底 + 灰描边 + 墨字，右上角一枚 Mono 小标"自建"。选中态统一：金环（外圈 --acc-live，内圈垫 --bg 形成间隙）。添加卡用虚线框，永远排在最后。
自带: --code-bg 底 + #E8A33D 字 · 52px · r 15px 自建: --panel 底 + --line 描边 + --ink-2 字 + 角标 Mono 7.5px 选中: box-shadow 0 0 0 2.5px --bg, 0 0 0 4.5px --acc-live
应用内的界面归谁管？
看板只管"壳"（触发条 + 面板 + 网格），iframe 内的应用界面由各应用自治，但要求共用同一份 token 文件 —— 把 shell.css 的 :root 变量块抽成 tokens.css，board-apps/*.html 各自引用，保证音乐/笔记/健身的按钮、颜色、圆角和壳一致。

05

场景三 · 管理和监控

信息权重排序 + 异常分级 + 设备集群的扩展形态。mockup 在场景一的机器里（切到"运行"Tab）。

信息多但权重不同，怎么排？
整页按权重自上而下重排：健康总结条 → 模型 → 定时任务 → 通道 → 权限 → 设备集群。V1 的 PROCESS 卡（pid/内存/指令数）降级为导航栏副标题的一行 Mono 小字 —— 开发者才看 pid，用户只关心"是不是都好"。权限从网格改为横滚小方块，沉底。
顺序: health → model → cron → channel → perm → cluster pid/uptime: 收进 nbar .sub · 不再独占卡片
怎么区分"扫一眼没问题"和"需要处理的异常"？
两级告警：页首健康条是总闸 —— 正常时绿底"一切正常"+ 各项计数，异常时红底 + 精确摘要"1 模型离线 · 1 任务失败" + 跳转按钮；异常单元自己发光 —— 出问题的模型/任务卡片描红边、内部给重试入口，其余卡片保持安静。正常 = 绿点呼吸（2s），异常 = 红点急呼吸（1s），扫一眼就知道几个红。
健康条: 绿 #E9F7EE / 红 #FDECEE · 圆点呼吸频率区分缓急 异常卡: box-shadow 0 0 0 1.5px --err-dot + 重试胶囊 · 其余卡片零干扰
未来多台设备接入，怎么展示设备集群？
每台设备是一个设备卡：深色分组头（设备名 + THIS/REMOTE 标签 + IP + 呼吸状态点），卡内按 MODEL / CRON 两行列出该设备的模型与任务。本机永远排第一。这个结构是列表的自然延伸，不需要新的布局范式 —— 设备多时就是一列设备卡，异常设备同样红缘。
设备卡: r-lg · 头 --panel-2 + 状态点 · 行 Mono k/v 结构 扩展: N 台设备 = N 张卡纵向排列 · 异常设备 红缘 + 头内告警

四类核心状态 × 四个关键界面。空状态永远带下一步动作，错误状态永远带重试入口。

风险与兜底
  • backdrop-filter 兼容性 — Android WebView 76+ 完整支持。兜底：@supports not (backdrop-filter:blur(1px)) 时 --glass 回退为 0.96 不透明度实色，功能无损。
  • 毛玻璃性能 — 模糊层只用于固定区域（导航/底栏/Sheet），不随内容滚动，GPU 开销恒定。低端机若掉帧，关掉 --blur 即可，视觉降级为半透明。
  • JetBrains Mono 缺字 — 中文会回退到 Noto Sans SC，V1 已如此，无新增风险。建议 Mono 区只放 ASCII 数据。
  • 深色模式 — 本规范只覆盖浅色。--code-bg / 看板沉浸底已是深色单元，可作为暗色主题的起点，单独立项。
