## 背景：MOV 是什么

MOV 是一个**开源 Android 应用**（MIT 协议）：在手机上配置多个大模型（DeepSeek、通义千问、OpenAI、Ollama 等，填 API Key 即可），然后在"房间"里给它们派活。

和普通 AI 聊天应用的本质区别：**它不只聊天，还能动手**。手机里有一个 Agent 引擎（叫 AgentLoop），AI 说的话会变成真实动作——

- **写文件**：每个房间是一个文件仓库，AI 产出的文档/代码直接落盘，每次覆盖自动留历史版本（快照）
- **执行设备指令**：开关手电筒、调亮度音量、朗读文字（TTS）、发通知、读写剪贴板等 30 个能力
- **定时任务（Cron）**：可以设定"每天早上 7 点自动做某事"，手机到点自己干活
- **多模型议会（Council）**：同一个问题同时问房间里的多个模型，互相评审、投票、汇总
- **HTML 打包成 APK**：AI 写的网页可以一键打包成签名 APK 安装包，装上手机就是一个真 App

两道安全闸（已写进架构，不可关闭）：
- **计划闸**：AI 动手前先交行动计划，人点"批准"才执行
- **写入预览**：AI 每次写文件前弹出内容预览卡，人点"确认写入"才落盘

## 〇、一句话定位

**MOV = 手机上的 AI 交付机器。用户说人话目标，稳定收到一个能用的东西。**

不是聊天应用，不是 skill 库，不是 prompt 合集。

## 一、宪法（两条，直接执行）

### 第 1 条：交付四问

每个功能上线前必须逐条回答，任何一条答不上就不上：

| # | 问题 | 反例（答不上的情形） |
|---|------|---------------------|
| 1 | **收到** — 产物真的到达用户手上了吗？ | AI 说"写好了"，文件躺在用户找不到的目录里 |
| 2 | **打得开** — 用户能打开它吗？ | 生成了 APK 但安装按钮是坏的 |
| 3 | **用得上** — 内容/格式/场景是对的吗？ | 菜谱写得很好，但用户要的是能点单的 App |
| 4 | **下次还在** — 明天还能找到、能继续改吗？ | 重启 App 后订单记录全没了 |

### 第 2 条：反对方针

| 不做 | 原因 |
|------|------|
| ❌ **Skill 大杂烩** | 曾有一份评估（MOV-CN-SCENARIOS.md）列出 32 个"可做场景"（菜谱、旅行规划、简历优化等）。结论：这些都是 prompt，聊天应用都能做且做得更好，做成清单是"没有骨架的模块清单"——追求花样，没有追求本质 |
| ❌ **假菜单** | 菜单上的菜厨房必须会做。例如"深度研究"需要联网抓取工具（还没做），把它写进菜单就是花架子 |
| ❌ **多场景并行** | 一次只打穿一个场景。十个铺着的场景不如一个"下单必达"的场景 |
| ❌ **本地大模型/向量数据库** | 重活全在云端 API，手机只发文本。加本地模型意味着 2-8GB 内存，不划算 |

## 二、失败经济学（与交付四问同级）

**交付机器的失败路径和成功路径一样重要。** 2026-07-23 真机 e2e 的现场：任务失败后界面只显示一枚失败徽章，状态卡死看起来像还在跑——用户既不知道发生了什么，也不知道已产出的部分去哪了、接下来能干嘛。这就是失败策略缺位的代价。

每个场景在设计时必须和交付四问**一起**回答失败三问：

| # | 问题 | 要求 |
|---|------|------|
| 1 | **用户看到什么？** | 失败原因必须说人话（"模型 API 超时"而不是一枚徽章）；状态必须终态化——成功/失败/已停止三选一，**禁止假卡死**（看着像在跑实际已死） |
| 2 | **部分产物归谁？** | 失败前已写入的文件默认保留在房间并明确列出（"已完成 2/5 步，这两个文件可用"），不许静默丢弃也不许假装没发生 |
| 3 | **重试怎么办？** | 分两期：**v1 穷人版续跑**——保留产物 + 人话原因 + 一键重开；重开时把"已完成 N 步、这些文件已存在"写进新任务的目标里，大脑看到 file.list 里文件已在，计划自然从"修改"而不是"重写"开始（file.list 本来就在计划上下文里，零后端改动）。**v2 真断点续跑**（循环状态序列化 + 恢复协议 + 已批准计划的重新授权）是独立大 feature，排期另议 |

**token 价值公式（含失败视角）**：

```
用户价值 = 交付完成率 × 产物质量
失败损失 = 已烧 token − 部分产物残值
```

失败经济学就是把失败损失里的"部分产物残值"找回来，并让重试成本最小。

## 三、MOV 的循环（主从结构，不是并列）

### 本体：生产循环（Agent + 文件产物）

给一个目标，交付文件/网页/APK，带版本历史。**MOV 的一切价值都从这一环长出来。**

交付质量的第一杠杆是**执行者看不看得见东西**（上下文/transcript 的完整性）：2026-07-23 e2e 血淋淋的证明——干活的脑子拿不到正确上下文时，交付就是废品，任何评审都救不回来。

### 放大器：自动循环（Cron + Agent）

生产循环 + 定时触发 = 手机在你睡觉时干活。聊天应用的结构盲区（它们永远被动等你开口）。

**自动循环也必须过交付四问，其中"收到"的语义单独定义**（这是自动循环成立的命门）：

| 四问 | 自动循环的答案 |
|------|---------------|
| 收到 | **三通道同时投递**：① 系统通知栏（锁屏可见，点击直达房间）② 房间未读角标 ③ 结果文件落盘房间仓库 |
| 打得开 | 点通知直接进到产出文件/消息 |
| 用得上 | 定时任务产出物格式固定、可预期（如"每日简报.md"） |
| 下次还在 | 每日产出按日期归档，历史可查 |

### 可选增强：验证循环（Council 多模型互审）

**不是本体，是生产循环的可选质检环节。** 评审团给一个"瞎子打工仔"投票投不出好交付——先保证执行者看得见东西，再谈多人评审。无评审配置时（reviewer 为空）应静默跳过，代码现状也是如此。重要的场景（合同审查类）再开。

## 五、试金石场景：烧烤摊点单 APK

**场景描述（说给没接触过这个项目的人）**：

一个烧烤摊老板，对手机上的 MOV 说一句话："帮我做个点单应用：羊肉串 5 块、板筋 3 块、烤韭菜 8 块、啤酒 6 块……"。几分钟后，他手机上多出一个叫"烧烤摊点单"的 App：顾客能看菜单、点菜、看到总价；下单后弹出老板的收款码；老板能看今天卖了多少单、收了多少钱、什么菜最好卖。想加菜，再说一句"加个烤茄子 10 块"，应用原地升级。

**为什么选它做第一个**：

1. **大众都懂**——不需要向任何人解释这是什么
2. **厨房真会做**——单文件 HTML + 浏览器存储，不需要联网工具、不需要后端服务器、不需要支付牌照（顾客扫老板自己的微信/支付宝收款码，钱不过平台）
3. **交易闭环完整**——菜单→挑选→付钱→（烧烤场景"送餐"就是端上桌，天然消解）
4. **压测最薄弱环节**——HTML→APK 是新功能，正好拿实战检验

## 七、场景扩展方法论（任何新场景过这五步）

1. **归类循环**：它是生产循环直出，还是生产循环 × Cron（自动循环）？
2. **如果是交易类场景，画三节骨架**：需求→收钱→交付，还是 需求→交付→收钱？节数不变，顺序由场景定（自动售货机买水=先付后拿；烧烤=吃完结账）。**注意：这只是交易类场景的启发式，不是 MOV 的骨架**——凌晨自动跑的研究简报没有"收钱"环节，非交易场景跳过本步。
3. **标模块挂载点**：用户将来要的扩展功能各挂在哪一节/哪一环？挂不上的需求砍掉。
4. **过交付四问**：收到/打得开/用得上/下次还在——哪问答不上就不做。自动循环按第三章的专用语义答"收到"。
5. **过失败三问**：用户看到什么/部分产物归谁/续跑还是重开——答不上就不做。
6. **查厨房**：菜单上每道菜现有工具做得出吗？做不出就不上菜单。

预筛过的候选（按优先级）：

| 候选 | 一句话 | 循环 |
|------|--------|------|
| 每日出题打卡 | 每天定时给孩子出口算题/古诗，错题自动累积成册 | 自动 |
| 数据看板 | 丢进一个 Excel/CSV，生成能看的图表网页 | 生产 |
| 合同审查 | 租房/劳动合同粘贴进来，多模型交叉标出风险条款 | 生产×验证增强 |
| 体检报告问答 | 体检 PDF 放进房间，"这个箭头什么意思"直接问 | 生产 |
| 小游戏工厂 | "给 6 岁孩子做个拼音小游戏" → 打包成 APK 发家人群 | 生产 |

## 附录 A：术语表

| 术语 | 含义 |
|------|------|
| **房间（Room）** | MOV 里的项目空间 = 聊天区 + 文件仓库。每个房间可配置不同的 AI 模型成员 |
| **AgentLoop** | MOV 的 Agent 引擎：把"目标"变成 计划→批准→执行→交付 的自动循环 |
| **transcript** | Agent 执行过程的上下文记录（计划、步骤、结果）。执行者拿不到完整 transcript = "瞎子打工"，交付必废 |
| **计划闸（Plan Gate）** | AI 动手前必须交计划，人批准才执行 |
| **写入预览** | AI 每次写文件前弹内容预览卡，人点"确认写入"才落盘 |
| **交付卡** | 任务完成时聊天区出现的结果卡片，列出产出文件，带「安装」等操作按钮 |
| **Council / 议会** | 多模型并行讨论：同一问题问房间所有模型，互相评审后汇总 |
| **Cron** | 定时任务系统（基于安卓 WorkManager），如"每天 7 点出题" |
| **快照** | 文件每次被覆盖前自动保存历史版本，可一键回滚 |
| **app.package** | 把房间里的 HTML 文件打包成签名 APK 安装包的工具 |
| **稳定包名** | APK 的包名按"房间+文件名"哈希生成，同一应用重复打包会覆盖升级而不是装出第二个图标 |
| **token** | 大模型 API 的计费单位，烧 token = 花钱 |
| **21770d7d** | 测试手机的设备编号（adb 设备 ID） |
| **e2e** | 端到端测试（从用户说一句话到产物装上手机的全流程） |
| **MOV-CN-SCENARIOS.md** | 此前对 113 个开源 AI 应用模板的中国场景筛选文档（32 个候选），本策划案将其降级为"参考池"而非承诺清单 |

## 附录 B：2026-07-23 打包链路修复清单

| # | 断点 | 修法 |
|---|------|------|
| 1 | 安装按钮可能指向错误文件（字符串切分解析，文件名含特定字符即坏） | 改为结构化 JSON 返回 |

| 5 | AI 完成任务后用户不知道要点「安装」 | 交付提示中明示 |
| 6 | 极端文件名（纯 ".html"）导致应用名为空 | 兜底"未命名应用" |
